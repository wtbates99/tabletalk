# Ecommerce — tabletalk Example Project

This is a complete, runnable example of a **tabletalk** project.
It demonstrates the core concept: **tabletalk is dbt for agents**.

```
┌─────────────────────────────────────────────────────────┐
│  dbt analogy                                            │
│                                                         │
│  contexts/*.yaml   ≈  sources.yml  (what data to expose)│
│  manifest/*.txt    ≈  manifest.json (compiled schema)   │
│  tabletalk apply   ≈  dbt compile  (regenerate output)  │
│  tabletalk query   ≈  running a model (agent query)     │
└─────────────────────────────────────────────────────────┘
```

---

## Quick start (no API key needed — uses Ollama locally)

```bash
# 1. Install Ollama  →  https://ollama.com
#    Pull the default model (excellent SQL generation, 7B params):
ollama pull qwen2.5-coder:7b

# 2. Install tabletalk with DuckDB support
pip install "tabletalk[duckdb]"

# 3. Create the database
python seed.py

# 4. Compile context definitions → manifests
tabletalk apply

# 5. Start an agent session (interactive CLI)
tabletalk query

# 6. Or launch the web UI
tabletalk serve
```

> **Switch to a cloud LLM later:** edit `tabletalk.yaml` and uncomment the
> `openai` or `anthropic` block — then `export OPENAI_API_KEY=sk-...` and
> you're done. No other changes needed.

## Project structure

```
ecommerce/
├── tabletalk.yaml          # Project config (database + LLM)
├── seed.py                 # Creates and seeds ecommerce.duckdb
│
├── contexts/               # Agent context definitions
│   ├── customers.yaml      # "Customers" agent — customer profiles & value
│   ├── sales.yaml          # "Sales" agent    — orders, revenue, products
│   ├── inventory.yaml      # "Inventory" agent — stock levels & alerts
│   └── marketing.yaml      # "Marketing" agent — campaigns & attribution
│
└── manifest/               # Compiled manifests (generated by `tabletalk apply`)
    ├── customers.txt
    ├── sales.txt
    ├── inventory.txt
    └── marketing.txt
```

## Agent contexts explained

Each `.yaml` file in `contexts/` defines one **agent context** — a scoped
view of the database that controls what an agent can see and query.

### customers.yaml
```yaml
name: customers
description: "Customer profiles, account status, and lifetime value."
datasets:
  - name: main
    tables:
      - name: customers
        description: "Registered customer accounts..."
```
Grants access to: `customers` only.

**Sample questions:**
- How many active customers do we have?
- Which cities have the most customers?
- Show customers who signed up in January 2024
- Who are our top 10 customers by lifetime value?

---

### sales.yaml
Grants access to: `orders`, `order_items`, `customers`, `products`, `categories`.

**Sample questions:**
- What is total revenue this month?
- Which products drive the most revenue?
- Show average order value by city
- What is the month-over-month revenue trend?
- Who are the top 5 customers by spend?

---

### inventory.yaml
Grants access to: `inventory`, `products`, `categories`.

**Sample questions:**
- Which products are below their reorder point?
- What is total inventory value at cost?
- Show all discontinued products that still have stock
- Which category has the highest inventory value?

---

### marketing.yaml
Grants access to: `campaigns`, `campaign_conversions`, `customers`.

**Sample questions:**
- What is the ROI of each campaign?
- Which channel drives more revenue — email or social?
- How many conversions did the New Year Sale generate?
- Which customers converted from multiple campaigns?

## Recommended Ollama models for SQL

| Model | Pull command | Notes |
|-------|-------------|-------|
| `qwen2.5-coder:7b` | `ollama pull qwen2.5-coder:7b` | **Default** — top-tier SQL, 7B params |
| `qwen2.5-coder:14b` | `ollama pull qwen2.5-coder:14b` | Larger, more accurate, needs 10 GB+ RAM |
| `codellama` | `ollama pull codellama` | Code-specialist, strong SQL |
| `llama3.2` | `ollama pull llama3.2` | Fast general-purpose model |
| `mistral` | `ollama pull mistral` | Strong at structured output |
| `phi3` | `ollama pull phi3` | Tiny and fast, decent SQL |

Change the model in `tabletalk.yaml`:
```yaml
llm:
  provider: ollama
  model: qwen2.5-coder:14b   # ← swap here for more accuracy
```

## The terraform analogy

Like Terraform, tabletalk lets you declaratively define and **redeploy** agents:

```bash
# Edit a context definition
vim contexts/sales.yaml

# Recompile — regenerates the manifest (like terraform apply)
tabletalk apply

# The agent now uses the updated schema
tabletalk query
```

The manifest acts as the "desired state" of what the agent knows.
Rerunning `tabletalk apply` reconciles the manifest with the current schema.

## Connecting to a real database

To connect to a production database instead of DuckDB:

```bash
# Interactive wizard
tabletalk connect

# Or import from an existing dbt profiles.yml
tabletalk connect --from-dbt my_dbt_project --target prod
```

Then update `tabletalk.yaml`:

```yaml
# Use a saved profile instead of inline credentials
profile: my_prod_profile
```
